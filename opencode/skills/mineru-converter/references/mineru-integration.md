# MinerU Integration

[MinerU](https://github.com/opendatalab/MinerU) is an open-source document
parser that converts PDFs into structured Markdown with extracted tables
(HTML), formulas (LaTeX), and figures. We build and run the official
`mineru-api` FastAPI server in a container, on port 8000.

## 1. Container setup

`scripts/setup_containers.sh` builds a local image tagged
`localhost/mineru-api:local`. Two variants:

| Variant | Image size | Backends | Use when |
|---------|------------|----------|----------|
| `slim` (default) | ~11 GB | `pipeline` only | Default for literature review - OmniDocBench 85+ already, no VLM cost |
| `full` (upstream) | ~35 GB | `pipeline` + `vlm-*` + `hybrid-*` + http-clients | You need hybrid-auto-engine for very complex tables/equations |

```bash
bash scripts/setup_containers.sh                      # slim build (default)
MINERU_VARIANT=full bash scripts/setup_containers.sh  # upstream Dockerfile
MINERU_REGION=china bash scripts/setup_containers.sh  # full + aliyun mirror
MINERU_REBUILD=1    bash scripts/setup_containers.sh  # force rebuild
SKIP_MINERU=1       bash scripts/setup_containers.sh  # no-op on CPU-only hosts
```

### Slim Dockerfile (in-script, default)

Generated by the `write_slim_dockerfile` function. Contents:

- Base: `nvidia/cuda:12.4.1-runtime-ubuntu22.04` (2.3 GB, runtime libs only)
- System: Python 3.11, fonts-noto-*, libgl1, libglib2.0-0, libsm6, libxext6, libxrender1
- pip: `mineru[core]>=3.0.0` (no `vllm`, `sglang`, `lmdeploy`, `flashinfer`)
- Models: `mineru-models-download -s huggingface -m pipeline` (skip 1.2B VLM)

### Why `pip install mineru[core]` and not `mineru[all]`?

The `[all]` extra pulls vLLM, sglang and lmdeploy as transitive dependencies
to enable VLM inference - over 4 GB of CUDA wheels we never use. `[core]`
covers the pipeline backend: PaddleOCR, UniMERNet (formula), PP-DocLayoutV2
(layout), table classifiers, basic OCR for 109 languages.

### Expected timings (first run, slim variant)

| Stage | Time |
|-------|------|
| Pull `nvidia/cuda:12.4.1-runtime` | 1-3 min (2.3 GB) |
| `apt install` system libs + fonts | < 1 min |
| `pip install mineru[core]` | 4-7 min (~6 GB downloads) |
| `mineru-models-download -m pipeline` | 1-3 min (~2 GB) |
| **Total slim build** | **8-15 min** |

`MINERU_VARIANT=full` adds the upstream `vllm/vllm-openai:v0.11.2` base
(~13 GB compressed download alone) and pulls all model bundles. Plan for
30+ minutes.

Subsequent invocations short-circuit the build and only run the existing
image. Use `MINERU_REBUILD=1` to force a rebuild after code or env changes.

GPU flag is detected at runtime:

| Runtime | Flag |
|---------|------|
| podman  | `--device nvidia.com/gpu=all` (CDI) |
| docker  | `--gpus all` |

## 2. Health probes

MinerU's `mineru-api` does not expose a dedicated `/health` route; the
Swagger UI at `/docs` serves as the liveness indicator once the app is
running.

```bash
python scripts/mineru_client.py health
# {"base_url": "http://localhost:8000", "reachable": true}

curl http://localhost:8000/docs
```

After `podman run`, the first request typically waits 30-90 s while
model weights are loaded into GPU memory. The setup script waits up to
5 minutes before giving up.

## 3. API used

| Endpoint | Body | Returns |
|----------|------|---------|
| `POST /file_parse` | multipart `files[]`, `lang_list`, `backend`, `parse_method`, flags | JSON `{backend, version, results: {<name>: {md_content, content_list, ...}}}` or `application/zip` when `response_format_zip=true` |
| `POST /tasks` | same as above | `202` with `task_id` + status/result URLs (async) |
| `GET  /tasks/{task_id}` | - | status payload |
| `GET  /tasks/{task_id}/result` | - | JSON or ZIP |
| `GET  /docs` | - | Swagger UI (used as health probe) |

`mineru_client.py` uses `POST /file_parse` (sync) only.

Key parameters (full list at `/docs`):

- `backend` (default in client: `pipeline`). Alternatives:
  - `hybrid-auto-engine` - highest accuracy + native text extraction
  - `vlm-auto-engine` - VLM model, high accuracy, English/Chinese only
  - `*-http-client` - offload VLM inference to a remote OpenAI-compatible server
- `lang_list` - OCR hint. English papers: `en`. CJK: `ch`. See the
  `/docs` page for the full language table.
- `return_md=true` - always on (we consume the `md_content` field).
- `return_content_list=true` - structured content list JSON (by reading order).
- `return_images=true` + `response_format_zip=true` - extracts image
  crops into a ZIP alongside the markdown.
- `start_page_id` / `end_page_id` - slice long PDFs.

## 4. When to use MinerU (LLM judgement)

Decision is **made by the LLM at Phase 4**, not hardcoded. Invoke MinerU when:

- arXiv HTML is missing or 404 (older / withdrawn papers).
- The paper is heavy on tables / formulas / figures that the LLM cannot
  extract reliably from section-split HTML.
- Non-arXiv PDFs with Unpaywall / OA hits.

Skip MinerU for:

- arXiv papers where HTML is available and the paper is text-heavy
  (faster, no GPU cost).
- Abstract-only reads (Layer 2 has those already).

## 5. Output layout

```
$LITREV_WORKDIR/<paper-id>/
  paper.pdf                  # from pdf_fetch.py
  manifest.json              # pdf_fetch source / url / bytes
  mineru/
    <name>.md                # primary Markdown output
    <name>_content_list.json # (when return_content_list=true)
    images/*                 # (when zip mode + return_images=true)
```

`<name>` is the sanitized basename of the uploaded PDF (MinerU's
`normalize_task_stem`).

## 6. Performance

- One PDF at a time. MinerU loads several models into GPU memory; concurrent
  requests can OOM. The server's internal concurrency semaphore defaults
  to 1 on Apple Silicon and reads `MINERU_API_MAX_CONCURRENT_REQUESTS`
  on Linux.
- Typical extraction time per paper on one consumer GPU
  (RTX 3090 / 4090, `pipeline` backend): 30-90 s for a 10-20 page paper.
  `hybrid-auto-engine` and `vlm-auto-engine` add VRAM + latency for
  higher accuracy.
- Cache outputs: re-runs against the same `<paper-id>` should reuse
  `mineru/<name>.md` if present.

## 7. Troubleshooting

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| `health: false` right after run | Models still loading | Wait 1-2 min, retry |
| `health: false` after 5 min | GPU OOM or CUDA mismatch | `podman logs mineru` |
| HTTP 500 on `/file_parse` | backend not in allowed set | Use `pipeline` (broadest support) |
| Empty `<name>.md` | Scanned image-only PDF, no OCR text | MinerU should OCR automatically - if it fails, mark paper as `mineru_failed` and rely on abstract |
| Build fails at `pip install` | Network to pypi.org blocked | `MINERU_REGION=china` switches to aliyun mirror + modelscope models |
| Build pulls wrong arch | multi-arch manifest mismatch | Dockerfile is arch-neutral; host must match base image support (x86_64 or aarch64) |

## 8. Known limitations

- Multi-column layouts, equations embedded in text, and rotated tables
  can degrade.
- The `pipeline` backend is hallucination-free but loses some structure
  on complex scientific papers; VLM-backed backends trade accuracy for
  GPU cost.
- No section-level metadata is emitted beyond what MinerU detects; the
  skill does **not** re-run `section_splitter.py` on MinerU output.
- Image OCR language set is broad (109 languages) but low-resolution
  scans still fail.

## 9. Why we build from source

The erikvullings/mineru-api community image we used in the very first
Stage B draft was delisted from Docker Hub, and its old `/convert/zip`
+ `/convert/markdown` API is gone. MinerU 3.0 (2026-03) introduced the
new `/file_parse` + `/tasks` endpoints we now target. Building from
the upstream `docker/<region>/Dockerfile` guarantees we track the same
API surface as the project's own release.
